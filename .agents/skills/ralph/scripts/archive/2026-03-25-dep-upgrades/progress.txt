## Codebase Patterns
- Prisma v7 uses generated client at `~/generated/prisma/client` (no `.js` extension) for Next.js/Turbopack imports
- Standalone scripts (outside Next.js) use relative paths like `../generated/prisma/client.js` with `tsx` runner
- PrismaClient in v7 requires an adapter (PrismaPg from `@prisma/adapter-pg`) - cannot be instantiated without one
- The `pnpm build` includes a `prebuild` step that runs `blog:index` - any Prisma import changes must work there too
- `esbuild` must be in `onlyBuiltDependencies` for `tsx` to work
- Turbopack resolves `~/generated/prisma/client` (without .js) but NOT `~/generated/prisma/client.js`
- ESLint enforces `@typescript-eslint/no-floating-promises` — use `void main()` for top-level async calls
- Zod 4: use `z.unknown().transform().pipe()` instead of removed `z.preprocess()`
- `@t3-oss/env-nextjs` v0.13+, `@hookform/resolvers` v5+, and `@trpc/*` v11.13+ all use Standard Schema (Zod 3/4 compatible)
- Auth.js v5: `export const { auth, handlers, signIn, signOut } = NextAuth({...})` — `auth()` replaces `getServerSession()`
- Auth.js v5: `AUTH_SECRET` and `AUTH_URL` env vars replace `NEXTAUTH_SECRET`/`NEXTAUTH_URL` — auto-detected by Auth.js
