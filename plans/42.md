# Plan: #42 — Unblock the build (decouple blog indexing from install/build)

**Branch:** `feat/42-unblock-build` · **Worktree:** `../wt-42` · **Base:** `feat/blog-search-rebuild` · Wave 1 (no deps)

## Goal

`pnpm install`, `pnpm lint`, `pnpm typecheck`, and `pnpm build` all succeed with a bogus/unreachable `DATABASE_URL` and no DB access on install. This is the prefactor that unblocks #43/#44/#45.

## Vertical slice

Config-only change across two source files + `package.json` + README docs. No new runtime logic, no schema change, no test runner (route manifest `○` vs `ƒ` is the verification artifact — all three specialists agreed no runner is warranted).

## Root cause (empirically verified by specialists)

- `package.json:18` `"prebuild": "pnpm blog:index"` writes to prod Postgres on every `pnpm build`.
- `src/app/sitemap.tsx:5` `export const dynamic = "force-static"` + `await api.blog.getAll()` at line 23 → **the sole build failure**: `next build` with unreachable DB fails `P1001 Can't reach database server` while prerendering `/sitemap.xml`.
- Prisma/pg-adapter is **lazy** — module-level `new PrismaClient()` in `src/server/db.ts` opens no connection at import; only real queries connect. So importing `~/trpc/server` is build-safe.
- `src/env.js:10` `z.string().url()` validates **syntax only**, not reachability — the bogus URL passes without `SKIP_ENV_VALIDATION`. No env change needed (and env is #44's surface — do not touch).
- **`projects/page.tsx` is ALREADY dynamic** (`ƒ /projects` in manifest): `api.projects.getAll()` → `src/trpc/server.ts` calls `headers()` (a Dynamic API) in `createContext`, auto-bailing every tRPC-server-caller page out of static generation. The issue's "known anchor" calling it statically-prerendered is **false**; the fix there is defensive/documentary, not a bug fix.

## Risks + mitigations

- **node_modules drift** → run `pnpm install` before verifying, or unrelated failures mask the real signal.
- **Per-request DB cost** on now-dynamic `/sitemap.xml` → negligible for a personal-site sitemap (crawler traffic only); `/blog` and `/projects` already do this. Not a new risk class.
- **Pre-existing `auth()` DB coupling** — `createTRPCContext` calls `auth()` (database session strategy) on every tRPC-server-caller invocation. Out of scope for #42; note in the final PR body so per-request DB work isn't mistaken as introduced here.
