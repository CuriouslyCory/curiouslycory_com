# Plan: #43 — Full-text blog search, end-to-end

**Branch:** `feat/43-fulltext-search` · **Worktree:** `../wt-43` · **Base:** `feat/blog-search-rebuild` (post-#42 tip) · Wave 2 (depends on #42, merged)

## Goal

Real full-text search on `/blog?q=` matching post **bodies**, not just titles. Extract prose+code from each post's `page.tsx` via the TS AST into `Post.content`, add a generated `tsvector` column + GIN index, and rewrite `getAll`'s `q`-branch to Postgres FTS — composing with existing tag/date/featured filters and pagination, output shape unchanged (UI untouched). Structure the search path so #44's semantic layer slots behind the same `getAll` entry point.

## Vertical slice

New DB-free extraction util (+ Vitest, the repo's first test runner) → indexer writes real `content`+`contentHash` and excludes `template/` → prisma migrate adoption (baseline `0_init` + generated tsvector migration) → raw-SQL FTS in `getAll`. Build must still pass with unreachable DB (no #42 regression).

## Critical guardrail

**Never touch production Postgres during implementation.** The indexer and migrations are verified against a **throwaway Docker Postgres** only. The one-time prod baseline (`migrate resolve --applied 0_init`) is a documented deploy-time operator step for the human, listed in the PR body — NOT run here.

### 2. AST extraction util — `src/lib/blog-content-extractor.ts` (new, DB-free, network-free)

Signature:
```ts
export interface ExtractSearchableTextOptions { includeAttributes?: string[]; } // default ["alt","title"]
export function extractSearchableText(source: string, options?: ExtractSearchableTextOptions): string
```
- Parse with `ts.createSourceFile("post.tsx", source, ts.ScriptTarget.ES2022, /*setParentNodes*/ true, ts.ScriptKind.TSX)` — **TSX ScriptKind is required** or JSX won't parse.
- Recursive `ts.forEachChild` walk, pushing into a `chunks: string[]` array (NEVER raw-concatenate — see spacing note):
  - `ts.isJsxText` → push `node.text`.
  - `ts.isJsxExpression` with `.expression` → collect `StringLiteral`/`NoSubstitutionTemplateLiteral` `.text`; for `TemplateExpression` push `head.text` + each `templateSpan.literal.text` (skip the interpolated `${}` expression code). This one path covers both `{" "}` spacers **and** `CodeBlock`'s `{\`...\`}` children — no special-casing for CodeBlock (confirmed: `CodeBlock` children are template literals, `fastify-telegram-bot/page.tsx:189`).
  - `ts.isJsxAttribute` whose name ∈ `includeAttributes` → push string value, supporting both `alt="..."` (StringLiteral) and `alt={"..."}`/`alt={\`...\`}` (expression) forms.
  - Continue `forEachChild` after each branch so nested JSX in expressions is walked.
  - **Deliberately NOT collected**: plain StringLiterals outside JSX-expression position (`export const metadata = {...}`, `className`, `href`, `src`) — keeps title/excerpt from being double-counted and excludes styling/URLs. The `/* ---bm ... --- */` frontmatter comment is trivia (never visited) — no regex strip needed.
- **Join with a space**: `chunks.join(" ")` then decode then collapse. Rationale is load-bearing — commit `ea7802a` fixed a *rendering* bug from JSX whitespace collapse (`{" "}` spacers between `<strong>`/inline tags in `introducing-infinite-docs/page.tsx`). The extractor must not depend on that collapse; forcing a separator makes `<strong>word</strong>{" "}next` and `<strong>word</strong> next` normalize identically. **This is a required test case, not an edge case.**
- **Entity decode** — hand-rolled (no `entities`/`he` in tree; not needed). Handle named (`amp lt gt quot apos nbsp mdash ndash hellip lsquo rsquo ldquo rdquo` — `&mdash; &apos; &quot;` confirmed in-repo) and numeric (`&#8212;`, `&#x2014;`); unknown entities pass through untouched (never throw).
- Final: `return decodeHtmlEntities(chunks.join(" ")).replace(/\s+/g, " ").trim();` (decode before collapse so `&nbsp;` folds).

### 6. Second migration `<ts>_add_post_content_hash_and_search_vector` (hand-written)

No shadow DB → hand-write the migration (it's small and needs the GENERATED clause hand-added regardless). Create `prisma/migrations/<timestamp>_add_post_content_hash_and_search_vector/migration.sql`:
```sql
ALTER TABLE "Post" ADD COLUMN "contentHash" TEXT;

-- Explicit ::regconfig cast: to_tsvector(text) is STABLE and rejected in a generated
-- column; to_tsvector(regconfig, text) is IMMUTABLE. STORED = auto-recomputed on write.
ALTER TABLE "Post" ADD COLUMN "searchVector" tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english'::regconfig, coalesce("title", '')), 'A') ||
    setweight(to_tsvector('english'::regconfig, coalesce("excerpt", '')), 'B') ||
    setweight(to_tsvector('english'::regconfig, coalesce("content", '')), 'C')
  ) STORED;

CREATE INDEX "Post_searchVector_idx" ON "Post" USING GIN ("searchVector");
```
Identifiers: no `@map`/`@@map` in schema → Prisma quotes `"Post"`,`"title"`,`"excerpt"`,`"content"`,`"searchVector"` exactly as declared. Optionally sanity-check the timestamp/name against what `prisma migrate diff --from-migrations prisma/migrations --to-schema-datamodel prisma/schema.prisma --script` would emit for the plain columns (that diff form needs a shadow DB, so prefer hand-writing).

### 8. Search — `getAll` in `src/server/api/routers/blog.ts` + `searchPostIds` helper

Add helper (keep it a named export / separate function so #44 can compose against it — this is the seam):
```ts
async function searchPostIds(db: PrismaClient, q: string): Promise<Array<{ id: string; rank: number }>> {
  return db.$queryRaw(Prisma.sql`
    WITH query AS (
      SELECT CASE WHEN numnode(websearch_to_tsquery('english', ${q})) = 0
                  THEN plainto_tsquery('english', ${q})
                  ELSE websearch_to_tsquery('english', ${q}) END AS tsq
    )
    SELECT "Post"."id" AS id, ts_rank_cd("Post"."searchVector", query.tsq) AS rank
    FROM "Post", query
    WHERE "Post"."published" = true AND "Post"."searchVector" @@ query.tsq
    ORDER BY rank DESC
  `);
}
```
- `Prisma.sql` (exported as `sql` from `~/generated/prisma/client`) binds `${q}` as a parameter — injection-safe.
- `numnode(...) = 0` → fallback to `plainto_tsquery` (handles stopword-only + stray operator chars); `@@` on empty tsquery matches nothing cleanly.
- `published = true` enforced **in SQL** (defense-in-depth) AND again via the Prisma `where` below.
- Returns the **full ranked id list** (no LIMIT/filters in SQL) — this is deliberate: full lists are what RRF needs (#44 seam), and at ~13 posts fetching all is trivial.

`getAll` q-branch:
```ts
const rankedIds = q ? await searchPostIds(ctx.db, q) : null;
// where: existing published/featured/tag/year composition PLUS:
...(rankedIds ? { id: { in: rankedIds.map(r => r.id) } } : {}),
// fetch: non-search keeps skip/take; search fetches the composed set then reorders+slices in JS
const [rows, totalCount] = await Promise.all([
  ctx.db.post.findMany({ ...query, ...(rankedIds ? {} : { skip, take: perPage }) }),
  ctx.db.post.count({ where: query.where }),
]);
const posts = rankedIds ? reorderByRank(rows, rankedIds).slice(skip, skip + perPage) : rows;
```
- `reorderByRank`: Map-based sort by `rankedIds` order (`findMany({id:{in}})` doesn't preserve order).
- **Pagination total when `q` present = `count({ where })` on the fully composed where** (FTS ∩ tag ∩ date ∩ featured) — corrects the issue's literal "count of FTS matches" (both engineers flagged this; overcount otherwise when filters active).
- Output shape `{ posts, pagination:{ total, page, perPage, totalPages } }` unchanged. `postSearchParamsSchema` unchanged.

## Decisions to record for PR sign-off (no docs/adr dir — capture in PR body)

1. Adopt `prisma migrate` via zero-downtime baselining (`migrate diff --from-empty` → `migrate resolve --applied 0_init`), retiring `db push`. Per-environment (prod, staging, each dev's local DB) one-time step. No shadow DB provisioned; workflow deliberately avoids `migrate dev`.
2. FTS via a STORED generated `tsvector` column (weighted title A/excerpt B/content C) + GIN index — not a trigger, not app-computed, not external search infra. Prevents content/vector write-skew.
3. `contentHash` added on `Post` now as a general "content changed" guard, deliberately shared with #44's re-embed logic (write in #43, read in #44 — no collision; hex sha256 over extracted prose).
4. Search entry point `searchPostIds(db,q) → {id,rank}[]` returns full ranked lists so #44 can RRF-fuse without rewriting `getAll`.
5. Pagination total with `q` = count of the fully composed filter set, not raw FTS hits.
